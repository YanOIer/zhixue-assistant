## Git工作流规范

### 分支策略
```
main (保护分支，只能PR合并)
  └── develop (开发分支)
        ├── feature/frontend-login (前端功能分支)
        ├── feature/backend-auth (后端功能分支)
        └── feature/rag-enhance (AI模块优化)
```

### 提交规范
```bash
# 格式：<type>: <description>
git commit -m "feat: 添加用户登录页面"
git commit -m "fix: 修复文档上传失败问题"
git commit -m "docs: 更新API文档"
```

**类型说明：**
- `feat`: 新功能
- `fix`: Bug修复
- `docs`: 文档更新
- `refactor`: 代码重构
- `test`: 测试相关

### PR模板
```markdown
## 改动说明
- 实现了xxx功能
- 修复了xxx问题

## 测试情况
- [ ] 本地测试通过
- [ ] 与后端/前端联调通过

## 相关Issue
Closes #123
```

## 应急处理

### 如果进度落后
1. **砍掉非核心功能**：优先保证MVP可用
2. **增加资源**：申请服务器、API额度等
3. **简化方案**：用简单实现替代复杂方案

### 如果组员遇到问题
1. **30分钟原则**：自己尝试30分钟无法解决则求助
2. **文档优先**：先查文档，再问组长
3. **知识共享**：解决问题后更新到文档
